## 一、这个任务到底要我们做什么？

题目给我们一个**半成品**的 ROS 工作空间，里面有一个叫 `demo_action` 的功能包。
包里定义了一种"动作"（Action）叫 `AddInts`，意思是"**把整数加起来**"。

任务要求我们补全两个 Python 文件：

- `action_server.py`（服务端）：**接收一个数字 N，计算 1+2+3+...+N 的和，每算一步把进度告诉客户端，算完返回最终结果**。
- `action_client.py`（客户端）：**把目标 N=100 发给服务端，然后一边收进度反馈、一边等最终结果**。

听起来就是个"累加求和"的小学题，为什么搞这么复杂？**因为重点不是累加，而是演示 ROS 的 Action 通信模式**。在真实机器人里，"导航到某个点"、"机械臂运动到目标位姿"、"打开门"这类任务都和累加一样：

- 耗时较长（不是一问一答就完事）；
- 过程中需要持续汇报进度（让用户/上层程序知道现在到哪一步了）；
- 可能需要中途取消；
- 最后要给出一个明确的结果。

所以这个看似幼稚的累加例子，本质是在演练"长耗时任务"的通信套路。

## 二、为什么需要 Action？它和 Topic、Service 有什么区别？

ROS 里有三种主要的通信方式，可以用"**点外卖**"来类比：

| 通信方式 | 类比 | 特点 | 典型场景 |
|---------|------|------|---------|
| **Topic（话题）** | 餐厅门口的喇叭广播："今日特价" | 单向、谁都能听、持续发 | 传感器数据流（雷达、相机、IMU） |
| **Service（服务）** | 你问收银员"还有座位吗？"，对方答"有" | 一问一答、同步、**短任务** | 一次性查询、触发某个瞬时动作 |
| **Action（动作）** | 你下单后，APP 显示"商家接单 → 骑手取餐 → 距您 500m → 已送达" | 一问多答、**异步**、可反馈可取消 | 导航、机械臂、长耗时任务 |

**关键差别**：Service 像电话——问一句答一句就挂了；Action 像外卖订单——下完单你不用一直占线，APP 会主动给你推送进度，最后告诉你送达了。

Action 把一次"长任务"拆成了三段信息：

```
        ┌─────────────┐
客户端 →│   Goal      │→ 服务端   "我要做这件事"（下单）
        └─────────────┘
        ┌─────────────┐
服务端 →│  Feedback   │→ 客户端  "现在做到这一步了"（进度推送，可多次）
        └─────────────┘
        ┌─────────────┐
服务端 →│   Result    │→ 客户端  "做完了，结果是这个"（一次性，任务结束）
        └─────────────┘
```

记住这三段，下面的代码就很好懂了。

## 四、逐文件详解

### 4.1 `AddInts.action` —— 通信契约

**文件内容**：

```
# 目标值
int32 num
---
# 最终结果
int32 result
---
# 连续反馈
float64 progress_bar
```

**讲解**：

`.action` 文件用三个减号 `---` 把内容分成三段，分别对应 Goal / Result / Feedback。
字段类型用的是 ROS 内置消息类型（`int32`、`float64`）。

| 段 | 字段 | 类型 | 含义 |
|----|------|------|------|
| Goal（目标） | `num` | int32 | 客户端要算 1+2+...+num |
| Result（结果） | `result` | int32 | 最终的累加和 |
| Feedback（反馈） | `progress_bar` | float64 | 进度（0.0 ~ 1.0） |

**编译时会发生什么？**
`catkin_make` 看到 `.action` 文件后，会自动生成一堆对应的 Python/C++ 消息类，比如 `AddIntsAction`、`AddIntsGoal`、`AddIntsResult`、`AddIntsFeedback`。这就是为什么代码里能直接 `from demo_action.msg import *`。

### 4.2 `action_server.py` —— 服务端

```python
#! /usr/bin/env python
# -*- coding: utf-8 -*-
import rospy
import actionlib
from demo_action.msg import *

class MyActionServer:
    def __init__(self):
        # 创建 Action 服务端
        self.server = actionlib.SimpleActionServer(
            "add_ints_action", AddIntsAction, self.cb, False)
        self.server.start()
        rospy.loginfo("Action 服务端已启动，等待目标...")

    def cb(self, goal):
        # 1. 获取目标并初始化变量
        num = goal.num
        sum_val = 0
        rate = rospy.Rate(1)  # 以 1Hz 的频率进行累加和反馈
        rospy.loginfo("接收到目标数值：%d", num)

        # 循环累加并连续反馈
        for i in range(1, num + 1):
            sum_val += i
            fb = AddIntsFeedback()
            fb.progress_bar = i / num
            self.server.publish_feedback(fb)
            rate.sleep()

        # 返回最终结果
        result = AddIntsResult()
        result.result = sum_val
        self.server.set_succeeded(result)
        rospy.loginfo("计算完成，最终结果：%d", sum_val)


if __name__ == "__main__":
    rospy.init_node("action_server")
    server = MyActionServer()
    rospy.spin()
```

**逐段讲解**：

**① 头部**（1~5 行）
- `#! /usr/bin/env python`：告诉系统用 Python 解释这个文件，配合 `chmod +x` 后就可以被 `rosrun` 直接调用。
- `from demo_action.msg import *`：导入 catkin 根据 `.action` 自动生成的消息类。

**② `__init__`：注册服务端**（8~12 行）
```python
self.server = actionlib.SimpleActionServer(
    "add_ints_action",     # ① Action 名字（客户端必须用同一个）
    AddIntsAction,         # ② Action 类型
    self.cb,               # ③ 收到 goal 时调用的回调函数
    False)                 # ④ 是否自动启动（False 表示稍后手动 start）
```
**关键点**：服务端的"名字"就像外卖店铺的店名 —— 客户端必须找对店名才能下单。

**③ `cb(goal)`：核心业务逻辑**（14~33 行）

这是整个服务端最重要的函数，**每收到一个 goal 就被调用一次**。流程：

1. **取目标 + 初始化**：`num = goal.num` 把客户端传来的数字读出来。
   `rospy.Rate(1)` 创建一个 1Hz 的"节拍器"，配合 `rate.sleep()` 实现"每秒执行一次循环"。
2. **循环累加 + 发布反馈**：
   ```python
   for i in range(1, num + 1):
       sum_val += i                              # 累加
       fb = AddIntsFeedback()
       fb.progress_bar = i / num                 # 进度 = 当前/总数
       self.server.publish_feedback(fb)          # 把进度推给客户端
       rate.sleep()                              # 等 1 秒再继续
   ```
   **`publish_feedback` 就是"推送进度"**，对应外卖 APP 里那个进度条。
3. **返回最终结果**：
   ```python
   result = AddIntsResult()
   result.result = sum_val
   self.server.set_succeeded(result)             # 告诉客户端："完成了，这就是结果"
   ```
   **`set_succeeded` 就是"订单送达"**，调用后任务结束。

**④ 主程序**（36~39 行）
- `rospy.init_node("action_server")`：向 ROS master 注册一个节点。
- `MyActionServer()`：实例化会触发 `__init__`，启动服务。
- `rospy.spin()`：让主线程持续等待，不被退出。

### 4.3 `action_client.py` —— 客户端

```python
#! /usr/bin/env python
# -*- coding: utf-8 -*-
import rospy
import actionlib
from demo_action.msg import *

# 最终结果回调
def done_cb(state, result):
    if state == actionlib.GoalStatus.SUCCEEDED:
        rospy.loginfo("计算完成，最终结果: %d", result.result)

# 服务激活回调
def active_cb():
    rospy.loginfo("服务端已激活...")

# 连续反馈回调
def fb_cb(fb):
    rospy.loginfo("当前进度: %.2f%%", fb.progress_bar * 100)

if __name__ == "__main__":
    rospy.init_node("action_client")

    # 1. 创建 Action 客户端
    client = actionlib.SimpleActionClient("add_ints_action", AddIntsAction)
    client.wait_for_server()  # 等待服务端启动
    rospy.loginfo("服务端已连接，准备发送目标...")

    # 2. 组织目标并发送
    goal = AddIntsGoal()
    goal.num = 100  # 设置目标数值，例如计算 1 到 100 的累加

    # 发送目标，并绑定三个回调函数
    client.send_goal(goal, done_cb=done_cb, active_cb=active_cb, feedback_cb=fb_cb)

    rospy.spin()
```

**逐段讲解**：

**① 三个回调函数**（7~18 行）

客户端是"被动"的 —— 它不主动轮询，而是**告诉服务端"有什么事就回调我"**：

| 回调 | 何时触发 | 我们做什么 |
|------|---------|-----------|
| `active_cb()` | 服务端开始处理 goal 时 | 打印一句"已激活" |
| `fb_cb(fb)` | 每次服务端 `publish_feedback` 时 | 打印进度百分比 |
| `done_cb(state, result)` | 服务端 `set_succeeded` 时 | 打印最终结果 |

**这就是 Action 比 Service 强的地方**：Service 只能阻塞等结果，Action 让你在等待期间能持续接收到信息。

**② 主程序**（20~35 行）

```python
client = actionlib.SimpleActionClient("add_ints_action", AddIntsAction)
client.wait_for_server()    # 阻塞等待，直到服务端上线
```
注意这里的 `"add_ints_action"` 必须和服务端完全一致，否则一辈子连不上。

```python
goal = AddIntsGoal()
goal.num = 100
client.send_goal(
    goal,
    done_cb=done_cb,
    active_cb=active_cb,
    feedback_cb=fb_cb)
```
**`send_goal` 是非阻塞的** —— 发完立刻返回，后续通知全部通过三个回调进入。

最后 `rospy.spin()` 让进程别退出，否则回调还没来得及收到程序就结束了。

### 4.4 `package.xml` —— 包的身份证

```xml
<buildtool_depend>catkin</buildtool_depend>
<build_depend>actionlib</build_depend>
<build_depend>actionlib_msgs</build_depend>
<build_depend>roscpp</build_depend>
<build_depend>rospy</build_depend>
<build_depend>std_msgs</build_depend>
```

`package.xml` 声明"**我这个包依赖了哪些 ROS 组件**"。
我们用了 `actionlib`（提供 SimpleActionServer/Client），所以必须列上，否则编译会找不到。

### 4.5 `CMakeLists.txt` —— 构建配置

需要重点理解的三段：

```cmake
find_package(catkin REQUIRED COMPONENTS
  actionlib actionlib_msgs roscpp rospy std_msgs)

add_action_files(FILES AddInts.action)        # 告诉 catkin：我有 Action 要生成
generate_messages(DEPENDENCIES actionlib_msgs std_msgs)

catkin_install_python(PROGRAMS
  scripts/action_server.py
  scripts/action_client.py
  DESTINATION ${CATKIN_PACKAGE_BIN_DESTINATION})   # 让 rosrun 能找到这俩脚本
```

**关键**：`add_action_files` + `generate_messages` 是让 `.action` 文件被自动编译成消息类的必备两步，少了任何一步都会出现"`AddIntsAction` 找不到"的错误。

## 七、常见问题排查

| 报错 / 现象 | 原因 | 解决 |
|------------|------|------|
| `Unable to register with master node` | 没启动 roscore | 先开一个终端跑 `roscore` |
| `ModuleNotFoundError: No module named 'demo_action.msg'` | 没 source 环境 | `source devel/setup.bash` |
| `AddIntsAction` 找不到 | `.action` 文件没被编译 | 检查 `CMakeLists.txt` 是否有 `add_action_files` 和 `generate_messages` |
| 客户端一直卡在 `wait_for_server` | 服务端名字不一致 / 没启动 | 核对两边的 `"add_ints_action"` 字符串 |
| `[rosrun] Permission denied` | 脚本没有执行权限 | `chmod +x scripts/*.py` |
| 客户端发完立即退出，看不到回调 | 缺 `rospy.spin()` 或被异常中断 | 检查脚本末尾 |
| 路径里 `work_2_2_ws` 重复但 `catkin_make` 报找不到 src | 你在外层目录运行 | `cd` 进**内层** `work_2_2_ws` 再编译 |

## 八、知识点总结与延伸

### 这次学到了什么？

1. **ROS Action 的三段式契约**：Goal / Feedback / Result，写在 `.action` 文件里。
2. **服务端套路**：`SimpleActionServer` → 在回调里 `publish_feedback` → `set_succeeded`。
3. **客户端套路**：`SimpleActionClient` → `wait_for_server` → `send_goal` + 三个回调。
4. **catkin 工作空间的构建流程**：`catkin_make` + `source devel/setup.bash`。

### 还能怎么扩展？

- **支持取消**：客户端可以调 `client.cancel_goal()`，服务端在循环里检查 `self.server.is_preempt_requested()` 来响应。
- **多客户端**：把服务端的 `False`（最后一参数）改成 `True`（auto_start），并研究 `SimpleActionServer` 的并发行为。
- **真实机器人场景**：把循环体从"累加"换成"调用 Unitree G1 的运动接口让机器人走一步"，反馈里改报"已行走距离/总距离"，结果里报告"到达目标点"。**这就是这个教学示例通往真实机器人控制的桥梁**。
